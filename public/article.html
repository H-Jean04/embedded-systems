Optimisation Énergétique

Pourquoi l'optimisation énergétique en embarqué
ne se résume pas au hardware

Jean HOUEDOUGBE Étudiant Ingénieur Embarqué
26 Février 2026 8 min de lecture 1.2k vues

Dans l'univers des systèmes embarqués, la consommation d'énergie est souvent perçue comme un problème purement hardware. Choisir le bon régulateur de tension, dimensionner correctement les condensateurs, opter pour des composants low-power... Ces décisions comptent, évidemment. Mais elles ne suffisent pas. Le firmware que vous écrivez peut annuler tous vos efforts matériels — ou au contraire, multiplier l'autonomie par deux.

1. Le mythe du hardware "suffisant"

Il est tentant de déléguer toute la responsabilité énergétique au schéma électronique. Un STM32L4 en mode ultra-low-power consomme théoriquement quelques microampères en veille. Mais "théoriquement" est le mot clé. Dans la pratique, un firmware mal conçu peut maintenir le processeur actif en permanence, empêcher les périphériques de passer en veille, ou générer des cycles inutiles qui font grimper la consommation à des niveaux inacceptables.

La réalité est simple : le hardware définit le plancher de consommation possible, le firmware détermine où vous vous situez réellement.

💡
À retenir

Un STM32L4 peut descendre à 300 nA en mode Shutdown. Mais un firmware qui ne gère pas les modes de veille peut maintenir une consommation 10 000 fois supérieure.

2. Les modes de veille : votre première arme logicielle

Les microcontrôleurs modernes offrent plusieurs niveaux de veille. Sur STM32, on distingue typiquement le mode Sleep, Stop, Standby et Shutdown. Chaque niveau éteint davantage de blocs internes, réduisant la consommation mais augmentant le temps de réveil.

La stratégie consiste à utiliser le niveau de veille le plus profond compatible avec vos contraintes temps réel. Si votre système doit répondre en moins d'une milliseconde, le mode Shutdown est exclu. Mais si vous attendez un événement capteur toutes les 30 secondes, rien ne justifie de rester en mode actif entre deux mesures.

C — STM32 HAL 
// Entrée en mode Stop 2 - consommation ~2µA
void enter_low_power_mode(void) {
    // Désactiver les périphériques non nécessaires
    HAL_SPI_DeInit(&hspi1);
    HAL_UART_DeInit(&huart2);

    // Configurer le réveil sur RTC
    HAL_RTCEx_SetWakeUpTimer_IT(&hrtc, 30, RTC_WAKEUPCLOCK_CK_SPRE_16BITS);

    // Entrer en mode Stop 2
    HAL_PWREx_EnterSTOP2Mode(PWR_STOPENTRY_WFI);

    // Le code reprend ici après réveil
    SystemClock_Config(); // Reconfigurer l'horloge
}

3. La gestion de l'horloge : souvent négligée

Beaucoup d'ingénieurs débutants configurent leur microcontrôleur à sa fréquence maximale et n'y touchent plus. C'est une erreur majeure. La consommation dynamique d'un processeur est directement proportionnelle à sa fréquence d'horloge. Tourner à 80 MHz quand la tâche en cours n'en a besoin que de 4 MHz, c'est consommer 20 fois plus d'énergie inutilement.

  • Réduire la fréquence d'horloge pendant les traitements légers
  • Éteindre les périphériques inutilisés via les registres RCC
  • Éviter les boucles d'attente active — préférer les interruptions
  • Utiliser le DMA pour les transferts de données sans intervention CPU
  • Optimiser les algorithmes pour réduire le nombre de cycles d'exécution
⚠️
Piège fréquent

Après un réveil depuis le mode Stop, l'horloge revient à sa configuration par défaut (HSI 16 MHz). Oublier de rappeler SystemClock_Config() peut déstabiliser tous vos timings.

Conclusion

L'optimisation énergétique est un travail d'équipe entre hardware et firmware. Un bon ingénieur embarqué ne choisit pas l'un ou l'autre — il maîtrise les deux dimensions. Commencez par identifier les phases actives et inactives de votre système, appliquez les modes de veille appropriés, gérez votre horloge dynamiquement, et mesurez. Toujours mesurer. Un analyseur de puissance comme le OTII Arc ou le PPK2 de Nordic est un investissement indispensable.

✅
Résumé pratique

Hardware → définit le minimum possible. Firmware → détermine où vous en êtes réellement. Maîtrisez les deux.
